## 2. Почему именно такой порядок

Порядок выстроен по принципу **«сначала безопасное и изолированное, в конце рискованное и архитектурное»**, чтобы каждый следующий шаг переиспользовал уже отлаженную инфраструктуру предыдущего:

1. **Умный фильтр** – чисто клиентская фича, без изменения схемы данных и сервера. Даёт мгновенную пользу, прогревает работу с тулбаром и карточками. Нулевой риск для данных.
2. **«Выданное мной» по секторам** – точечный фикс существующего бага в серверном scope (`myCreated` сейчас игнорирует сектор). Изолированно, без схемы, переиспользует уже существующий выпадающий список секторов.
3. **Поиск** – добавляет первый новый серверный обработчик `InvokeTemplate` и панель поиска в тулбаре (рядом с фильтром из шага 1). Схему не трогает (ищет по существующим атрибутам).
4. **Вкладка «Заметки»** – первое добавление атрибута в шаблон `KanbanTask`. Изолированная фича, задаёт эталонный паттерн «новый атрибут → отдать в `GetTaskDetails` → сохранить в `SaveTask` → новая вкладка». Этот паттерн переиспользуют шаги 5–6.
5. **Отключение уведомлений** – добавляет атрибут (паттерн из шага 4) и вклинивается в путь уведомлений. Средний риск, потому что трогает рассылку.
6. **Повторяемые задачи** – несколько атрибутов + хук в жизненный цикл (`DoMoveTask` → Done) + опционально серверная фоновая задача. Самая объёмная из «делаемых».
7. **Несколько исполнителей** – помечено как «на подумать». В канбане статус привязан к карточке, поэтому полноценный мультиассайн ломает модель. Внедряется в последнюю очередь и в виде безопасного варианта «Соисполнители/Наблюдатели».

Каждый шаг самодостаточен – можно остановиться на любом и иметь рабочую систему.

## 3. Общая архитектура (нужно знать перед любым изменением)

Из блокнота `Kanban` + сверки с `scripts/`:

- **Бэкенд**: C#-скрипты в среде Soyuz-PLM BIS v3. Главный файл – `scripts/SOYUZ_UPLOAD_KanbanScreen_script.cs` (~5025 строк). Точка входа – `public override Object Invoke(String methodName, InfoObject obj, Object inputParams)`. Маршрутизация команд – `switch` по `methodName` (ориентир: строки **165–228**). **Новые серверные команды добавляются как новый `case` в этом switch.**
- **Шаблон задачи**: `scripts/SOYUZ_UPLOAD_KanbanTask_script.cs` (~207 строк) – `OnBeforeSave`, уведомление при смене `Assignee`.
- **Фронтенд (IE11 WebBrowser!)**:
  - `scripts/KanbanBoard_HTML.html` (~1475 строк) – разметка + Liquid. Карточки доски рендерятся **на сервере через Liquid** (4 блока: Todo/InProgress/Waiting/Done).
  - `scripts/kanban.js` (~3771 строк) – ES5-логика.
  - `scripts/kanban.css` (~3040 строк) – стили.
- **Мост JS → C#**: `window.external.InvokeTemplate("ИмяКоманды", параметры)`. Параметры – это **плоская строка** (поля через `|`) или массив строк. На сервере нормализуется `GetParamStr(inputParams)`, разбивается `ParsePipeArgs(raw, N)`.
- **Хранилище**: `InfoObject` (задача – шаблон `KanbanTask` в контейнере `All_Kanban_Tasks_Folder`), `DataContainer`, `NamedValue` (справочники `Ref_KanbanStatus`, `Ref_KanbanPriority`, `Ref_KanbanTags`), `WorkItem` (уведомления `WorkLoads\BASIC\Message\ExclamationKanban`). Комментарии и чек-листы – сериализованы в строковых атрибутах `CommentsJSON` / `SubtasksJSON`.

### Атрибуты шаблона `KanbanTask` (существующие)
`TaskName`, `TaskDetails`, `KanbanStatus` (NamedValue), `Priority` (NamedValue), `DueDate`, `OriginalDueDate`, `CompletedDate`, `ReturnCount`, `Assignee` (ссылка на User), `Creator` (stable key, строка), `IsPrivate` (bool), `Tags` (строка через запятую), `ChangeLog` (JSON), `SeenByList` (строка), `AttachedObjects`, `AttachedContainers`, `CommentsJSON`, `SubtasksJSON`, `SubtasksTotal`, `SubtasksDone`, `DirectumLink`.

## 4. Жёсткие правила (соблюдать во всех фичах)

**IE11 (`kanban.js`) – ЗАПРЕЩЕНО:** `let`, `const`, стрелочные функции `=>`, `fetch`, `Promise`, шаблонные строки `` `...` ``, `class`, `for...of`, деструктуризация.
- Любую функцию, вызываемую из inline `onclick`, объявлять как `window.имя = function (...) { ... };` **до** использования.
- Только `var`, обычные `function`, конкатенация строк через `+`.

**C# / PLM:**
- Перед записью атрибутов существующего объекта обычно нужен `task.GetEditable()` (см. эталон в `DoGetTaskDetails`, строка ~1872, где `SeenByList` пишется через `task.GetEditable()` + `editable.Save()`). В обработчиках записи (`DoMoveTask`, `DoSaveTask`) задача уже редактируемая – пишется напрямую `task["X"] = ...; task.Save();`.
- `NameKey` пользователя/объекта может быть пустым – не полагаться только на `??`. Стабильный ключ пользователя берётся как `NameKey` → `AccountId` → `Id` (см. функции `GetUserStableKey`, `FindUserByKeyOrNull`).
- Критичные проверки прав – **только на сервере** (`CanUserSeeTask`, `CanAssignUserInScope`). Клиентский UI прячет кнопки, но не является защитой.
- DotLiquid в BIS v3 ненадёжен на некоторых строковых сравнениях – использовать проверки на пусто/непусто.

**Добавление нового атрибута в шаблон `KanbanTask`** (из блокнота `Soyz-PLM`) – два пути:
1. **Через Конфигуратор (рекомендуется):** в модуле конфигурирования открыть шаблон `KanbanTask`, добавить подчинённый объект «Определение атрибута» (`AttributeDef`): задать `NameKey` (ключ), `DataType` и вариант отображения. Типы: `Text` (строка; первые 255 символов индексируются, остальное в полнотекст), `DateTime`, `Bool`, `LinkToUser`, и др.
2. **Через скрипт:** `var def = new AttributeDef(...); def.Save();` (см. блокнот `Soyz-PLM`, раздел про `AttributeDef`).

**Персональные настройки пользователя** (например, «не уведомлять глобально») – хранить в реестре БИС: `Service.SetUserRegistryValue<T>(path, value)` / `Service.GetUserRegistryValue<T>(path, default)`. **Не** использовать `PropertyBag` (живёт только в памяти сессии) и **не** `NamedValue` (это справочники).

## 5. Универсальная процедура внедрения одной фичи

1. **Бэкап.** Скопировать затрагиваемые файлы из `scripts/` в сторону (они хранятся как текст в PLM; в блокноте `Kanban` есть снапшоты `90–95` с SHA-256 для сверки).
2. **Открыть Конфигуратор** Soyuz-PLM, найти соответствующий скриптовый объект (экран `myKanbanTest`, шаблон `KanbanTask`, ресурсы `kanban.js` / `kanban.css` / `KanbanBoard_HTML.html`).
3. **Если фича требует атрибут** – сначала создать атрибут в шаблоне `KanbanTask` (раздел 4).
4. **Внести правки по инструкции из подпапки** – строго по якорям (имя метода/строка). Номера строк – ориентир на 09.06.2026; после первой правки они сместятся, поэтому всегда ищите по **сигнатуре метода / тексту-якорю**, а не только по номеру.
5. **Сохранить → дать Roslyn перекомпилировать** (ошибки компиляции видны в редакторе скриптов).
6. **Обновить экран доски** (F5 / кнопка «Обновить») – клиент перечитает `kanban.js` / HTML.
7. **Прогнать смок-тест** из подпапки.
8. После крупной фичи – обновить снапшоты `90–95` в базе знаний (по желанию).

## 7. Откат

Все изменения – это правки текстовых скриптов. Откат = вернуть сохранённую копию файла в Конфигураторе и перекомпилировать. Для фич с новым атрибутом: атрибут можно оставить (он не мешает), либо удалить определение атрибута из шаблона `KanbanTask`. Данные в уже заполненных атрибутах при удалении определения станут недоступны через API – поэтому при сомнениях атрибут лучше просто перестать использовать, а не удалять.
