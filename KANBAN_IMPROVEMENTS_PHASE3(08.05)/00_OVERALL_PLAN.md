## Общий контекст для агента

Платформа: **Soyuz-PLM**, веб-интерфейс рендерится в **WebBrowser-контроле (Trident, IE11 в режиме edge через `<meta http-equiv="X-UA-Compatible" content="IE=edge">`)**. Это означает:

- HTML5 Drag-and-Drop API работает (карточки задач уже используют `dragstart` / `dragover` / `drop` / `dataTransfer` — см. `kanban.js:166-231`).
- Современный JS поддерживается, но осторожно: `flex` через `display: -ms-flexbox`, `var()` НЕ использовать (CSS-переменные в IE11 не работают), стрелочные функции и `let/const` НЕ использовать в продовом коде Kanban (используется `function`/`var`).
- `JSON.parse` доступен, `Array.prototype.indexOf` доступен.
- `external` (он же `window.external`) — мост в C# через `InvokeTemplate("MethodName", "args")` (см. `KANBAN_IMPROVEMENTS_PHASE2/06_subtasks_checkboxes/plan.md` — образец паттерна).

Все JSON-атрибуты задач в БД — **простые строковые `Text`-атрибуты с MaxLength 1 МБ**. JSON парсится клонированным `ParseComments` / сериализуется `SerializeComments` в C# (см. `SOYUZ_UPLOAD_KanbanScreen_script.cs:3046+`).

Автор задачи хранится двумя путями:
1. **Системно**: `task.CreatedByUser` (тип `User`) — *для фильтрации лучше не использовать без индексации*.
2. **В кастомном атрибуте `Creator` (Text, NameKey текущего пользователя)** — пишется в `DoCreateTask` (line 379) и `DoCreateGroupTask` (line 475). **Этот атрибут — основной источник истины** для проверок прав и фильтра «Выданные мной».

## Порядок реализации (рекомендуемый)

Можно делать **независимо и параллельно**. Конфликты:

- №1 и №5 трогают модалку задачи (`tcmOverlay`) — но разные участки (вкладка «Чек-лист» vs поле «Описание» во вкладке «Основное»). Конфликтов нет.
- №2 и №4 трогают шапку (`navigationTitle` в `KanbanBoard_HTML.html`) — №4 добавляет кнопку рядом с «Мои задачи / Все задачи». Делать №2 после №4 (или согласовать стилистику в №2 с уже добавленной кнопкой из №4).
- №3 — изолированная правка в `kbFillUsers` / `kbRestoreViewMode`.

**Рекомендуемый порядок коммитов**: 5 → 3 → 1 → 4 → 2.

## Конвенции коммитов

Как в Phase 2:
- `feat(kanban): <что>` — новая фича
- `fix(kanban): <что>` — исправление
- `refactor(kanban): <что>` — рефакторинг без изменения поведения
- `docs(kanban): <что>` — документация
- `chore(kanban): pmszcfg <что>` — изменение конфигурации (ручное)

Если фича требует ручного изменения `.pmszcfg` (атрибуты в `KanbanTask`) — выделить отдельный коммит `chore(kanban): pmszcfg ...` и положить рядом `CONFIG_REQUIRED.md`. В этой фазе **новых атрибутов БД не требуется** — все правки укладываются в существующие `SubtasksJSON`, `Creator`, `KbViewMode` (PropertyBag).
