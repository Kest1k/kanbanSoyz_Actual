# KanbanTask — Скрипт шаблона задачи

**Файл:** `SOYUZ_UPLOAD_KanbanTask_script.cs`
**Тип:** C# скрипт шаблона объекта PLM

## Назначение
Скрипт шаблона объекта «Задача канбан-доски». Обрабатывает:
- Уведомления при назначении задачи (OnBeforeSave → WorkItem ExclamationKanban)
- Сброс SeenByList при смене исполнителя (для бейджа «НОВАЯ»)
- AJAX-запросы: CRUD задачи, комментарии, вложения, история

## Атрибуты шаблона KanbanTask

| Атрибут | Тип | Описание |
|---------|-----|----------|
| `КанбанНазвание` | string | Название задачи |
| `КанбанОписание` | text | Описание задачи |
| `КанбанСтатус` | int | Статус: 0-3 (колонка на доске) |
| `КанбанПриоритет` | string | `low`, `medium`, `high`, `urgent` |
| `КанбанСрок` | datetime | Дата дедлайна |
| `КанбанИсполнитель` | reference | Ссылка на пользователя-исполнителя |
| `КанбанСоздатель` | reference | Ссылка на пользователя-создателя |
| `КанбанДатаСоздания` | datetime | Дата создания задачи |
| `КанбанДатаЗавершения` | datetime | Дата перевода в статус «Готово» |
| `КанбанКомментарии` | text (JSON) | JSON-массив комментариев |
| `КанбанВложения` | text (JSON) | JSON-массив привязанных объектов |
| `SeenByList` | string | Ключи пользователей, просмотревших задачу (через запятую) |

## Уведомления (OnBeforeSave)

При смене исполнителя (Assignee) скрипт:
1. Сравнивает нового исполнителя с сохранённым (`PersistedValue`)
2. Если исполнитель изменился и это не самоназначение — создаёт `WorkItem` из шаблона `ExclamationKanban`
3. Очищает `SeenByList` — новый исполнитель увидит бейдж «НОВАЯ» на доске
4. В Subject передаёт: название задачи, имя отправителя (в родительном падеже), приоритет

**Шаблон уведомления:** `WorkLoads\BASIC\Message\ExclamationKanban` — кастомный дубликат Exclamation с RTF-форматированием и кликабельной ссылкой на доску.

## AJAX-обработчики

### `getCard` — Получение данных карточки
```
Параметры: taskId
Ответ: {
  "id": "...",
  "title": "...",
  "details": "...",
  "status": 0,
  "statusName": "Надо сделать",
  "priority": "medium",
  "dueDate": "15.01.2025",
  "assigneeId": "...",
  "assigneeName": "Иванов И.И.",
  "creatorName": "Петров П.П.",
  "createdDate": "10.01.2025",
  "completedDate": "",
  "isOwner": true,
  "canEdit": true,
  "canDelete": true
}
```

### `saveCard` — Сохранение изменений
```
Параметры: taskId, title, details, status, priority, dueDate
Действие:
  1. Валидирует входные данные
  2. Обновляет атрибуты объекта
  3. Если статус изменился на 3 (Готово) → ставит дату завершения
  4. Если статус изменился с 3 → очищает дату завершения
  5. Создаёт запись в истории изменений
Ответ: { "ok": true }
```

### `deleteCard` — Удаление задачи
```
Параметры: taskId
Проверки: Только создатель или руководитель
Действие: Удаляет объект из PLM
Ответ: { "ok": true }
```

## Комментарии (чат)

### `getComments` — Получение комментариев
```
Параметры: taskId
Ответ: {
  "comments": [
    {
      "id": "comment_1736500000000",
      "author": "Иванов И.И.",
      "authorId": "...",
      "initials": "ИИ",
      "text": "Текст комментария",
      "time": "10.01.2025 14:30",
      "isMine": true
    }
  ]
}
```

Комментарии хранятся в атрибуте `КанбанКомментарии` как JSON-массив.

### `addComment` — Добавление комментария
```
Параметры: taskId, text
Действие: Добавляет объект комментария в JSON-массив с timestamp-ID
Ответ: { "ok": true, "comment": { ... } }
```

### `deleteComment` — Удаление комментария
```
Параметры: taskId, commentId
Проверки: Только автор комментария может удалить
Действие: Удаляет комментарий из JSON-массива
Ответ: { "ok": true }
```

## Вложения

### `getAttachments` — Получение вложений
```
Параметры: taskId
Ответ: {
  "attachments": [
    {
      "id": "...",
      "objectId": "...",
      "name": "Деталь-001",
      "template": "Деталь",
      "type": "object",
      "date": "10.01.2025"
    }
  ]
}
```

Вложения хранятся в атрибуте `КанбанВложения` как JSON-массив ссылок на объекты PLM.

### `addAttachment` — Привязка объекта
```
Параметры: taskId, objectId, type ("object"/"container")
Действие:
  1. Проверяет существование объекта в PLM
  2. Проверяет, не привязан ли уже
  3. Добавляет запись в JSON-массив вложений
Ответ: { "ok": true, "attachment": { ... } }
```

### `removeAttachment` — Отвязка объекта
```
Параметры: taskId, attachmentId
Действие: Удаляет запись из JSON-массива вложений
Ответ: { "ok": true }
```

## История изменений

### `getRevisions` — Получение истории
```
Параметры: taskId
Действие: Возвращает историю изменений объекта из PLM (ревизии атрибутов)
Ответ: {
  "revisions": [
    {
      "date": "10.01.2025 14:30",
      "user": "Иванов И.И.",
      "changes": "Статус: «Надо сделать» → «В работе»"
    }
  ]
}
```

## Логика статусов

```
Статус 0 (Надо сделать) ──drag──→ Статус 1 (В работе)
Статус 1 (В работе)     ──drag──→ Статус 2 (Ожидание)
Статус 2 (Ожидание)     ──drag──→ Статус 1 (В работе)
Любой статус             ──drag──→ Статус 3 (Готово) → автоматически ставится completedDate
Статус 3 (Готово)        ──drag──→ Любой → очищается completedDate
```
